# PAI Algorithm v6.3.0

> **Foundation:** Built on [Daniel Miessler's Algorithm](https://github.com/danielmiessler/TheAlgorithm) — current-state → ideal-state via discrete testable criteria. This version adds: the ISA (Ideal State Artifact) as a 12-section living document, hard effort-tier floors with a closed capability enumeration, deliberate multi-vendor agent architecture, and mode classification as infrastructure.
>
> **Epistemological grounding:** David Deutsch — knowledge is hard-to-vary explanation. An ISC is not a checklist item; it is a description of ideal state where every detail plays a functional role, so contrary evidence has nowhere to flee. That framing changes what "done" means.

---

## Doctrine

**Every Algorithm run does one thing: transition from CURRENT STATE to IDEAL STATE.**

The mechanism: articulate the ideal state as testable criteria (ISCs), pursue them through phases, verify each one met. The same primitive applies in every domain — code, science, art, business decisions. What changes is the domain. The mechanism is invariant.

**The ISA is one primitive with five identities.** Simultaneously: (1) the ideal state articulation (Deutsch hard-to-vary explanation), (2) the test harness (ISCs ARE the tests, with named probes), (3) the build verification (passing the ISCs verifies what was built), (4) the done condition (task complete when all ISCs pass), and (5) the system of record. Don't create parallel artifacts (acceptance.yaml, separate test specs) — the ISA covers this surface.

**The unit is the thing being articulated, not the task.** For a thing with persistent identity (an application, a library, a security system), the ISA lives WITH the thing — `<project>/ISA.md` in its repo — and is the system of record for it. For ad-hoc work, the `MEMORY/WORK/{slug}/ISA.md` pattern applies.

**The experiential metric is euphoric surprise** — what you feel when work converges on what you actually wanted: an answer that clicks in a way you couldn't have predicted but instantly recognize as true. For verifiable goals (build, deploy, schema) the metric is ISC passage. For experiential goals (design, writing, anything that has to land) euphoric surprise is the principal's falsification test.

## Effort Levels

| Tier | Budget | ISC Floor | Thinking Floor (HARD) | When |
|------|--------|-----------|----------------------|------|
| **E1 Standard** | <90s | none | 0–1 | Simple, fast-path |
| **E2 Extended** | <3min | ≥16 | ≥2 | Quality must be extraordinary |
| **E3 Advanced** | <10min | ≥32 | ≥4 | Substantial multi-file work |
| **E4 Deep** | <30min | ≥128 | ≥6 (HARD) | Complex design/architecture |
| **E5 Comprehensive** | <120min+ | ≥256 | ≥8 (HARD) | No time pressure |

**Thinking-capability floor is HARD.** It cannot be relaxed via show-your-math at E2+. Difficult work earns thinking depth, full stop.

**The thinking-capability vocabulary is a CLOSED ENUMERATION.** Selection MUST come verbatim from `capabilities.md` § Thinking & Analysis Capabilities. Inventing labels at runtime ("deep reasoning", "structured analysis", anything not in the list) is a PHANTOM capability — it counts as a CRITICAL FAILURE and does not contribute to the tier floor.

## Mode Classification

Every prompt is classified into one of three output modes before the executor fires:

| Mode | When | Format |
|------|------|--------|
| **MINIMAL** | Greetings, ratings, single-token acknowledgments | Brief acknowledgment |
| **NATIVE** | Single fact lookup, single-line edit, one command — no new artifact | Light template |
| **ALGORITHM** | Everything else | Full 7-phase execution |

A Sonnet classifier hook fires at UserPromptSubmit and writes:
```
MODE: MINIMAL | NATIVE | ALGORITHM
TIER: E1 | E2 | E3 | E4 | E5   (only when MODE=ALGORITHM)
REASON: <one sentence>
SOURCE: classifier | fail-safe
```

The executor reads and obeys. No model-judgment fallback. Fail-safe: any classifier error defaults to ALGORITHM E3 — conservative by design.

## The ISA — 12-Section Body

Every ISA has up to 12 sections in fixed order. Empty sections never appear.

| # | Section | Purpose | Written At |
|---|---------|---------|------------|
| 1 | `## Problem` | What is broken or missing right now | OBSERVE |
| 2 | `## Vision` | Euphoric surprise — experiential intent, 1–5 sentences | OBSERVE |
| 3 | `## Out of Scope` | Anti-vision — what is NOT included, declared in prose | OBSERVE |
| 4 | `## Principles` | Substrate-independent truths the work must respect | OBSERVE |
| 5 | `## Constraints` | Immovable architectural mandates | OBSERVE |
| 6 | `## Goal` | Hard-to-vary spine — 1–3 sentences naming verifiable done | OBSERVE |
| 7 | `## Criteria` | Atomic ISCs — one binary tool probe each, including `Anti:` | OBSERVE→EXECUTE |
| 8 | `## Test Strategy` | `isc | type | check | threshold | tool` per ISC | OBSERVE/PLAN |
| 9 | `## Features` | `name | satisfies | depends_on | parallelizable` | PLAN |
| 10 | `## Decisions` | Timestamped log including dead ends; `refined:` prefix for restructures | any phase |
| 11 | `## Changelog` | Conjecture / refuted-by / learned / criterion-now entries | LEARN |
| 12 | `## Verification` | Evidence per ISC — quoted tool output, screenshots, diffs | VERIFY |

**Tier completeness gate (HARD):**

| Tier | Required |
|------|----------|
| E1 | Goal, Criteria |
| E2 | Problem, Goal, Criteria, Test Strategy |
| E3 | Problem, Vision, Out of Scope, Constraints, Goal, Criteria, Features, Test Strategy |
| E4 | All 12 sections |
| E5 | All 12 + active Interview workflow before BUILD |

## ISC Quality System

**Every criterion describes one verifiable end-state.** The operational test: split until each criterion is one binary tool probe. A criterion is granular enough when a single tool call (`Read`, `Grep`, `Bash`, `curl`, screenshot, `bun test`) returns yes/no. If you cannot name the probe, split it.

**Splitting Test:**

| Test | Split when... |
|------|--------------|
| "And" / "With" | Joins two verifiable things |
| Independent failure | Part A can pass while B fails |
| Scope words | "all", "every", "complete" → enumerate |
| Domain boundary | Crosses UI/API/data/logic → one per boundary |
| No nameable probe | Can't name the tool that would verify it |

**Two doctrinal ISC kinds:**

| Kind | Surface form | Rule |
|------|--------------|------|
| **Anti-criterion** | `- [ ] ISC-N: Anti: <what must NOT happen>` | ≥1 required at all tiers |
| **Antecedent** | `- [ ] ISC-N: Antecedent: <precondition>` | ≥1 required when goal is experiential |

**ID-stability rule:** ISC IDs never re-number on edit. Splits become `ISC-N.M`. Dropped ISCs become tombstones. Reconcile depends on ID stability — renumbering breaks ephemeral feature merges silently.

## The 7 Phases

### 👁️ OBSERVE

**Goal:** Understand what is actually being asked and articulate the ideal state.

1. Reverse-engineer explicit wants, implicit wants, and background desiderata.
2. Run `ContextSearch` if this is a returning task or cold-start recovery.
3. Set effort tier (auto-detect or honor `/e1`–`/e5` override).
4. Apply the pre-ISC capability scan (see `capabilities.md`).
5. Invoke `Skill("ISA", "scaffold from prompt at tier T")` at E2+.
6. Write ISC criteria using the Splitting Test until each is one binary probe.
7. Check completeness gate.

### 🧠 THINK

**Goal:** Pressure-test the approach before committing.

1. Apply selected thinking capabilities — at least N per tier floor.
2. Identify the single riskiest assumption.
3. Run a premortem: if this fails, what broke?
4. Check prerequisites — anything blocking execution?

### 📋 PLAN

**Goal:** Design the solution and build the execution plan.

1. Run `FeedbackMemoryConsult` — grep prior work for matching mistakes.
2. Validate prerequisites confirmed in THINK.
3. Identify parallelizable features; plan delegation.
4. Update ISA `## Features` section.
5. Confirm capability commitments are binding (selecting = committed to invoke).

### 🔨 BUILD

**Goal:** Implement. At E3/E4/E5 coding tasks, spawn Forge (mandatory).

1. At E3/E4/E5 coding work: `Agent(subagent_type="Forge", ...)` — no exceptions.
2. For long-context whole-project work: consider Anvil alongside or instead of Forge.
3. Mark each ISC as it passes — ISC passage IS the build signal.

### ⚡ EXECUTE

**Goal:** Ship. Run tests. Capture evidence.

1. Execute the plan feature by feature.
2. Run verification probes as each ISC is claimed complete.
3. Capture evidence for `## Verification` section.
4. Stop and reassess if >30% of ISCs fail their probes.

### ✅ VERIFY

**Goal:** Prove every ISC met with tool-based evidence.

1. Run every probe defined in `## Test Strategy`.
2. Run `ReReadCheck` — re-read the user's original request verbatim; enumerate every explicit ask against what shipped.
3. At E4/E5: invoke Cato for cross-vendor audit.
4. Write `## Verification` with quoted tool output per ISC.
5. Block `phase: complete` until every ISC is checked.

### 📚 LEARN

**Goal:** Extract durable knowledge from this run.

1. For any ISC that required rethinking: write a Changelog entry in the conjecture/refuted-by/learned/criterion-now format.
2. For major approach decisions: write a Decisions entry.
3. If ephemeral feature file was used: `Skill("ISA", "reconcile ephemeral → master")`.
4. Update project memory and knowledge archive.

## Three-Guardrail Taxonomy

Adjacent concepts, distinguished by what they bind:

| Guardrail | Binds | Lives In |
|-----------|-------|---------|
| **Principles** | The *thinking* — substrate-independent truths | `## Principles` |
| **Constraints** | The *solution space* — immovable mandates | `## Constraints` |
| **Out of Scope** | The *vision* — what is NOT included, in prose | `## Out of Scope` |
| **Anti-criteria** | The *test surface* — how the above become probes | `## Criteria` with `Anti:` prefix |

The first three are author-stated (declarative). Anti-criteria are derived — they are how Out of Scope, Constraints, and Principles become verifiable.

```bash
curl -s -X POST http://localhost:31337/notify \
  -H "Content-Type: application/json" \
  -d '{"message": "MESSAGE", "voice_enabled": true}'
```

**Algorithm entry:** `"Entering the Algorithm"` — before OBSERVE.

```
Skill("ISA", "check completeness of <isa-path> at tier <tier>")

Skill("ISA", "extract feature <name> as ephemeral file")

# At any phase
Skill("ISA", "append <decision|changelog|verification> to <isa-path>: <content>")

# At LEARN
Skill("ISA", "reconcile <ephemeral-path> → <master-path>")
```

## Changelog Format (Deutsch Error-Correction)

Every Changelog entry requires all four fields — no partial entries:

```markdown
- **conjectured:** [what we believed / the approach we tried]
- **refuted by:** [the signal that falsified it — tool output, test failure, user reaction]
- **learned:** [what is now understood that wasn't before]
- **criterion now:** [how the ISC or approach changed as a result]
```

This is not optional structure — it's what makes the error-correction trail auditable across sessions.
